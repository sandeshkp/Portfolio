What was once a half-day manual grind — a single analyst downloading raw data from SAP HANA, manually slicing it across states, categories and sub-categories, building Excel reports one by one, and emailing them individually to 22 different state teams — is now a fully automated, production-grade pipeline that wakes up before the business does, pulls fresh data, constructs over 100 distinct reports across every organisational dimension, and dispatches them to the right recipients through three mail servers — all within a single hour, every single working day, without a single human touch.

This is not a reporting tool. This is an intelligence distribution system built on a robust ETL backbone — SAP HANA OData services, Python batch automation, a scheduler-driven .py script execution layer, MySQL as the operational data store, WinSCP/FTP for early-morning data delivery, and SMTP integration across Zoho, Outlook and Google Workspace mail infrastructure used across the organisation.

Every working morning, before any decision-maker could look at a number, a member of the sales analytics team had to execute the following sequence manually:

❌ Before Automation

• Login to SAP HANA, navigate to report transaction, set date filters manually

• Download raw Excel dump — often 50,000+ rows across all states and SKUs

• Open Excel, apply pivot tables, slice by state, category, sub-category

• Build each state's report individually — 22 states × multiple tabs

• Add YTD, MTD, target vs. achievement, growth columns manually

• Name, format, and save each file separately

• Open Outlook, compose 22 individual emails, attach correct file to each

• Repeat the same for director-level consolidated PPT summary

• Entire process: 3.5 to 4.5 hours every single morning

✅ After Automation

• SAP HANA backend script pushes fresh data to FTP server at 6:00 AM

• Windows Task Scheduler triggers Python script at 6:30 AM

• Script validates data presence, cleans and loads into MySQL

• 100+ report files generated across all dimensions automatically

• State-specific reports routed to correct recipients via SMTP

• Director-level consolidated reports and PPT summaries dispatched

• Full audit log written — every report, every recipient, every timestamp

Root Cause of the Old Problem

• No structured data pipeline — SAP was used as a glorified download portal

• No single source of truth — each analyst's Excel had different filters and logic

• No report versioning — recipients had no way to verify if they had the latest file

• No scalability — adding a new state or report type required weeks of manual rework

• No audit trail — zero traceability on who received what, when, and with what data

System Architecture — Two Phases

Phase 1 — FTP-Based Architecture (Initial Deployment)

In the first phase, a custom extraction script was written directly inside the SAP HANA backend in collaboration with the IT team. This script was scheduled to execute at 6:00 AM every working day, extracting the prior day's complete sales transaction data and pushing the output file to a designated FTP server managed via WinSCP.

Phase 2 — OData ETL Architecture (Production Upgrade)

The FTP-based push was later replaced with a proper production-grade ETL pipeline using SAP HANA OData services. Scheduler Layer

Windows Task Scheduler executes the master .py script at a fixed trigger time each morning. The script is self-healing — it checks for data file presence before proceeding, retries on connection timeout, and writes all failures to a structured error log with alert mail dispatch to the admin.

Scheduler Config

• Trigger: Daily 06:30 AM on working days

• Script: master_report_runner.py

• On failure: Error log + admin alert mail

• Retry logic: 3 attempts with 5-min interval

Data Validation Gates

• File/API response presence check

• Row count threshold validation

• Date freshness check (yesterday's data)

• Null rate check on critical columns

ETL Pipeline — Step by Step

1
Extract — OData API Pull from SAP HANA
Authenticated GET requests to SAP HANA OData endpoints using session-based tokens. Data fetched in paginated chunks (1,000 rows per request) and assembled into a single raw Pandas DataFrame. Fields extracted include invoice date, dealer code, state, category, sub-category, SKU, pairs sold, MRP, net realisation, and target figures.
2
Validate — Schema and Freshness Checks
Automated checks confirm expected columns are present, null rates on critical fields are below threshold, and the maximum invoice date in the pull equals the previous working day. Any gate failure halts execution and dispatches an alert.
3
Transform — Clean, Enrich and Compute
State name standardisation (SAP codes → readable names), category hierarchy mapping, MRP normalisation, YTD / MTD / Prior Year aggregation columns computed using Pandas groupby and rolling window operations. Growth % and target achievement % derived and appended as computed columns.
4
Load — MySQL Upsert with Partitioning
Cleaned DataFrame upserted into MySQL using INSERT … ON DUPLICATE KEY UPDATE logic via SQLAlchemy. Table is partitioned by transaction date and state code to ensure query performance stays constant as historical data grows. A copy of the raw file is also archived in a dated folder for audit traceability.
5
Report Generation — 100+ Files in Parallel
Report engine queries MySQL views per report type, applies openpyxl formatting templates, generates Excel files with conditional formatting, summary rows, and chart sheets. Director-level consolidated reports and PowerPoint presentation decks generated using python-pptx with auto-populated data tables and charts.
6
Dispatch — SMTP Multi-Server Mail Engine
Report-to-recipient mapping loaded from a configuration table in MySQL. Each recipient's state-specific reports are bundled and dispatched through the appropriate mail server (Zoho for sales team, Outlook for management, Google Workspace for field force). Mail body includes summary statistics inline. Full dispatch log written per run.

Core Script Structure (Pseudocode)

# master_report_runner.py — Scheduled 06:30 AM daily import pandas as pd import numpy as np from sqlalchemy import create_engine from report_engine import ReportFactory from mail_engine import SMTPDispatcher from etl import ODataExtractor, DataValidator, MySQLLoader # ── Step 1: Extract ────────────────────────────────── extractor = ODataExtractor(base_url=SAP_ODATA_URL, auth=ODATA_CREDS) raw_df = extractor.fetch_incremental(since=yesterday()) # ── Step 2: Validate ───────────────────────────────── validator = DataValidator(raw_df) if not validator.run_all_gates(): SMTPDispatcher.send_alert("DATA VALIDATION FAILED", validator.errors) exit(1) # ── Step 3: Transform ──────────────────────────────── clean_df = ( raw_df .pipe(normalise_state_codes) .pipe(map_category_hierarchy) .pipe(compute_ytd_mtd_columns) .pipe(compute_growth_and_achievement) ) # ── Step 4: Load ───────────────────────────────────── loader = MySQLLoader(engine=create_engine(MYSQL_CONN)) loader.upsert(clean_df, table='sales_daily_fact') loader.archive_raw(raw_df) # ── Step 5: Generate 100+ Reports ──────────────────── factory = ReportFactory(engine=create_engine(MYSQL_CONN)) for report_config in REPORT_MANIFEST: factory.build(report_config) # Excel + PPT where applicable # ── Step 6: Dispatch ───────────────────────────────── dispatcher = SMTPDispatcher(config=MAIL_SERVER_CONFIG) dispatcher.dispatch_all( report_dir = OUTPUT_DIR, mapping_table= 'report_recipient_map', run_date = today() ) write_run_log(status='SUCCESS', reports_sent=factory.count, recipients=dispatcher.count)

Report Splitting Logic

Each base query result is split across three axes simultaneously using a single Pandas GroupBy operation — state, category and sub-category — producing a separate, correctly named output file per combination. File naming follows a standardised convention that encodes date, report type, state code and dimension level to enable downstream automation by recipients.

# State × Category × Sub-category split in one pass for (state, category, sub_cat), grp in df.groupby([ 'state_name', 'category', 'sub_category']): filename = f"{run_date}_{state}_{category}_{sub_cat}_sales.xlsx" write_report(grp, OUTPUT_DIR / filename, template='state_report') report_registry.register(filename, recipient=get_recipient(state))

Multi-Server SMTP Delivery Engine

The organisation uses three distinct mail ecosystems across different departments and seniority levels. The dispatch engine reads each recipient's assigned mail server from a configuration table and routes accordingly — ensuring delivery through the correct authenticated SMTP connection.

Dispatch Logic

# Mail dispatch with server routing for recipient in recipient_list: server_type = recipient['mail_server'] # 'zoho' | 'outlook' | 'google' attachments = get_reports_for(recipient['state'], recipient['level']) msg = MIMEMultipart() msg['Subject'] = f"{run_date} | Daily Sales Report — {recipient['state']}" msg.attach(MIMEText(build_summary_html(recipient), 'html')) for filepath in attachments: msg.attach(build_attachment(filepath)) smtp = SMTP_POOL[server_type] smtp.send_message(msg, to_addrs=recipient['email_list']) log_dispatch(recipient, attachments, status='SENT')

Mail Body — Inline Summary

Each automated mail includes an HTML summary table directly in the mail body — key KPIs (previous day sales, MTD achievement %, YTD growth vs. prior year) displayed inline so the recipient gets the headline numbers without opening the attachment. Attachments carry the full drill-down detail.

Audit and Error Handling

• Every dispatched mail logged to MySQL with timestamp, recipient, report list, server used

• SMTP failures caught and retried up to 3 times before being logged as failed

• Failed dispatches trigger an admin alert with full error trace

• Daily dispatch summary report emailed to pipeline owner at end of run
