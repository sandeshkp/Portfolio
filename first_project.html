The Production Planning Software (PPS) is a Flask-based internal web application developed to streamline the daily mould allocation and machine scheduling process at Walkaroo International. The system converts dealer requirements — pulled live from SAP HANA — into structured production schedules, assigning moulds and machines based on historical sales patterns, inventory levels, and production capacity.

Before PPS, production planners manually processed dealer order sheets in Excel, cross-referenced mould availability registers, and built machine schedules by hand — a process that consumed 3–4 hours each morning and was prone to scheduling conflicts and missed allocations. PPS reduced this to a single-click operation.

Key Objectives

• Automate daily mould and machine scheduling from live dealer demand data

• Integrate directly with SAP HANA OData APIs for real-time stock and order data

• Apply historical sales intelligence to prioritise high-velocity SKUs

• Provide planners with a visual schedule interface — not raw spreadsheets

• Containerise and deploy via Docker for reliable cross-environment operation

Core Challenge

The scheduling problem is fundamentally a constraint-satisfaction problem: given N dealer requirements, M available moulds, and K available machines — generate an optimal allocation that maximises production output while respecting mould-machine compatibility, shift capacity, and priority rules. The system handles this using a weighted priority queue built in Python, with compatibility matrices maintained in MySQL.

• Mould-to-machine compatibility constraints (not every mould fits every machine)

• Shift-based capacity limits per machine per day

• Dealer priority tiers (A / B / C classification from SAP)

• SKU velocity weighting from 12-month historical sales data

• Backlog carry-forward logic for unfulfilled prior-day requirements

The architecture is split into three distinct layers: data ingestion, scheduling computation, and the Flask delivery interface. This separation ensures that heavy ETL and scheduling logic never blocks the web interface.

Layer 1 — Data Ingestion

• SAP HANA queried via OData REST APIs at schedule generation time

• Dealer order data, mould master, and machine availability extracted

• Python ETL validates, cleans, and normalises incoming records

• Historical sales data loaded from MySQL (12-month rolling window)

• All raw extracts stored in staging tables before processing

Layer 2 — Scheduling Engine

• Weighted priority queue built per dealer × SKU × mould combination

• Compatibility matrix loaded from MySQL configuration tables

• Constraint solver iterates allocation respecting shift capacity limits

• Backlog engine carries forward unmet demand from previous day

• Output schedule written to MySQL output tables

Layer 3 — Flask Delivery Interface

• Flask reads final schedule from MySQL output tables

• Jinja2 templates render machine-wise and mould-wise schedule views

• Excel export generated via openpyxl for factory floor distribution

• Role-based access: planners vs. floor supervisors see different views

• Docker container ensures consistent deployment across environments

Deployment

The application is containerised using Docker with a docker-compose.yml configuration that orchestrates the Flask app container alongside the MySQL container. This enables one-command deployment on any server without dependency conflicts.

Data Pipeline — End to End

The PPS pipeline runs on-demand when a planner triggers schedule generation, and also has a scheduled pre-fetch that runs at 6:00 AM to pull overnight SAP HANA data so the morning schedule can be generated instantly.

Step 1 — SAP HANA Extraction

• OData API calls made to SAP HANA endpoints for dealer order tables

• Mould master data fetched (mould ID, dimensions, compatible article types)

• Machine availability status fetched from production module

• API responses parsed from JSON into Pandas DataFrames

• Extraction handles pagination for large order volumes (batch size 1000)

Step 2 — Cleaning and Validation

• Null checks on critical fields: dealer code, article code, quantity, mould ID

• Duplicate order line detection and de-duplication

• Quantity outlier detection — orders above 3σ flagged for planner review

• Article code standardisation (SAP codes mapped to internal naming convention)

• Invalid mould-article combinations rejected with logged error reason

Step 3 — Priority Scoring

Each dealer-SKU requirement is assigned a composite priority score before entering the scheduling engine. The score combines four weighted factors:

• Dealer tier weight (40%) — A/B/C classification from SAP dealer master

• Historical velocity weight (30%) — 12-month average sales for this SKU in this region

• Backlog weight (20%) — number of days this requirement has been unmet

• Stock urgency weight (10%) — days-of-stock remaining at current sell rate

Step 4 — Schedule Generation

• Requirements sorted by descending priority score

• For each requirement: compatible moulds fetched from compatibility matrix

• Available machines identified (not at capacity for current shift)

• Optimal mould-machine pair selected; allocation written to output table

• Shift capacity counters updated after each allocation

• Unallocated requirements written to backlog table for next day carry-forward

Step 5 — Output and Distribution

• Final schedule written to MySQL output tables (machine-wise view)

• Flask reads output tables and renders schedule on web interface

• Excel export generated with openpyxl — one sheet per machine

• Schedule summary emailed to production manager via SMTP automation

# Priority scoring — simplified Python logic def compute_priority_score(row, velocity_df, backlog_df): dealer_weight = {'A': 1.0, 'B': 0.7, 'C': 0.4} velocity = velocity_df.get(row['article_code'], 0) backlog_days = backlog_df.get(row['order_id'], 0) stock_urgency = 1 / (row['days_of_stock'] + 1) score = ( dealer_weight[row['tier']] * 0.4 + (velocity / velocity_df.max()) * 0.3 + (backlog_days / 30) * 0.2 + stock_urgency * 0.1 ) return round(score, 4)

Core Features

⚙️ Auto Schedule Generation

One-click schedule generation that pulls live data from SAP HANA, runs the priority-weighted allocation engine, and produces a complete machine-wise production schedule — in under 60 seconds.

📦 Mould Compatibility Matrix

A configurable mould-to-article and mould-to-machine compatibility matrix maintained in MySQL. Planners can update compatibility rules via the admin panel without touching code.

🔁 Backlog Carry-Forward

Unmet requirements automatically carry forward to the next day with an escalating backlog weight — ensuring high-backlog orders surface to the top of the next day's priority queue.

📤 Excel Schedule Export

Final schedules exportable as formatted Excel files with one sheet per machine, ready for floor supervisors. Generated via openpyxl with conditional formatting for priority indicators.

📊 Historical Velocity Weighting

12-month rolling average sales velocity per article per region is used to weight allocation priority — ensuring fast-moving SKUs are always scheduled ahead of slow movers when capacity is constrained.

🔐 Role-Based Access

Planners see the full schedule generation interface. Floor supervisors see a read-only machine-view. Admins can update compatibility matrices and capacity configurations. Flask session-based authentication.

⚠️ Conflict & Outlier Flagging

Orders with unusually high quantities, invalid mould-article combinations, or machine capacity overloads are flagged before the schedule is finalised — giving planners a review queue rather than a failed run.

🐳 Docker Deployment

Fully containerised with Docker Compose — Flask app + MySQL in isolated containers. Environment variables manage SAP HANA credentials and database configs. One-command startup: docker-compose up.

Flask Application Interface

The Flask application serves as the user-facing interface for the entire PPS system. It is designed for production planners who need a fast, no-friction tool — not a complex analytics dashboard. The UI is deliberately simple: one primary action (generate schedule), with secondary views for review and export.

Route Structure

• GET / — Login / session check → redirect to dashboard

• GET /dashboard — Main planner dashboard; shows today's schedule status

• POST /generate — Triggers SAP pull → ETL → scheduling engine → writes output

• GET /schedule — Renders today's generated schedule (machine-wise view)

• GET /export — Streams Excel file download of today's schedule

• GET /backlog — View unallocated requirements carried forward

• GET /admin — Mould compatibility matrix and capacity configuration (admin only)

Session and Authentication

• Flask session-based authentication with bcrypt password hashing

• Three roles: planner, supervisor, admin

• Route decorators enforce role-based access on sensitive routes

• Sessions expire after 8 hours (one shift duration)

Template Structure

• base.html — Shared layout with nav and session context

• dashboard.html — Schedule generation trigger + status cards

• schedule.html — Machine-wise schedule table with priority colour coding

• backlog.html — Unallocated orders table with backlog age indicators

• admin.html — Mould matrix editor and machine capacity configuration

Technical Learnings

• SAP HANA OData API rate limits require careful batch sizing and retry logic

• Constraint-satisfaction problems at production scale benefit from greedy weighted sorting over exhaustive solvers — speed matters more than theoretical optimality in an operational morning context

• Docker Compose significantly simplified deployment across dev / staging / prod

• Building an admin UI for the compatibility matrix was essential — without it, every mould/machine change would have required a code deployment
